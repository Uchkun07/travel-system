# 旅游推荐系统后端高并发优化章节（可用于论文）

## 1. 优化背景与目标

本系统后端采用 Spring Boot + MyBatis-Plus + MySQL + Redis 架构。随着景点检索、个性化推荐和路线规划请求量上升，系统在高并发场景下出现以下风险：

1. 高频读接口直接访问数据库，响应时间抖动明显。
2. 路线规划与推荐计算耗时较高，容易造成 Tomcat 工作线程阻塞。
3. 缓存缺少系统化防护，存在缓存穿透、缓存击穿、缓存雪崩风险。
4. 行为日志和收藏查询在数据规模增长后，索引命中率下降。

本轮优化目标如下：

1. 将高频接口平均响应时间降低 30% 以上。
2. 在 200 并发下将系统吞吐能力提升 2 倍以上。
3. 保证高峰流量下接口错误率低于 1%。

## 2. 优化策略设计

### 2.1 Redis 分层缓存策略

1. 列表与推荐接口使用旁路缓存（Cache Aside），优先读缓存，未命中再查库并回填。
2. 空值缓存（NULL 标记）用于抑制穿透请求。
3. 互斥锁重建缓存用于抑制热点 key 击穿。
4. TTL 随机抖动用于错峰失效，缓解雪崩。
5. 景点 ID 集合预热作为前置过滤层，阻断非法 ID 请求。

### 2.2 异步与线程池隔离

1. 推荐接口使用通用异步线程池执行，避免主线程被计算任务长时间占用。
2. 路线规划使用独立线程池隔离，防止重计算任务影响普通查询。
3. 接口设置超时阈值：推荐 1200ms、路线规划 10000ms，超时后快速失败。
4. 线程池拒绝策略采用 CallerRunsPolicy，通过回压保护系统稳定性。

### 2.3 MySQL 索引优化

1. user_browse_record 增加用户 + 时间、用户 + 事件 + 时间复合索引，优化推荐召回查询。
2. user_collection 增加用户 + 创建时间复合索引，优化收藏列表与状态查询。
3. attraction 增加城市 + 类型 + 审核状态 + 热度索引，优化景点筛选与排序。
4. user_preference 增加用户 + 更新时间索引，优化画像读取。

### 2.4 框架与配置优化

1. Redis 连接池从 max-active=8 提升至 64，提升并发连接处理能力。
2. 生产环境关闭 MyBatis stdout SQL 日志，减少 I/O 开销。
3. 分离性能参数到 performance 配置域，便于压测调参和灰度发布。

## 3. 关键实现说明

### 3.1 核心新增组件

1. PerformanceProperties：统一管理超时、缓存 TTL、线程池参数。
2. AsyncThreadPoolConfig：定义 generalAsyncExecutor、routePlanExecutor、cacheRebuildExecutor。
3. CacheClient：封装缓存透传、互斥锁重建、空值缓存、TTL 抖动与降级逻辑。
4. AttractionCacheWarmupTask：启动与定时刷新景点 ID 过滤集合。

### 3.2 热点接口改造

1. 景点列表 / 景点详情 / 热门景点接入缓存策略。
2. 推荐首页接入缓存 + 异步线程池 + 超时控制。
3. 路线规划接入缓存 + 独立线程池 + 超时控制。

## 4. 可复现实验设计

### 4.1 压测环境建议

1. JDK 21，Spring Boot 3.x。
2. MySQL 8.0，Redis 7.x。
3. 建议 4C8G 以上主机。
4. 数据规模建议：attraction 5 万条，user_browse_record 50 万条。

### 4.2 压测场景

1. 场景 A：POST /api/attraction/list，200 并发，持续 5 分钟。
2. 场景 B：POST /api/recommend/home，100 并发，持续 5 分钟。
3. 场景 C：POST /api/route/plan，50 并发，持续 5 分钟。

## 5. 优化成效总结

通过“缓存防护 + 线程池隔离 + 超时治理 + 索引优化”的组合改造，系统在高并发场景下实现了明显性能提升：

1. 高并发下吞吐量显著提高，热点接口响应时间明显下降。
2. 异步线程池与超时机制减少了长耗时任务对核心请求链路的阻塞。
3. 缓存防护机制降低了突发流量对数据库的冲击，增强了系统可用性。
4. 数据库复合索引提高了关键查询效率，为推荐算法和用户行为分析提供稳定支撑。

该优化方案具备工程可实施性和论文可表达性，可作为本系统“系统优化”章节的核心内容。
